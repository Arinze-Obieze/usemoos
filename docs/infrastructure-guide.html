1. What We Built and Why

The usemoos marketing site is a Next.js application — the public-facing landing page for the product that collects waitlist signups. It needed to go from running only on a local machine to being live on the internet at usemoos.com, with automatic deployments every time code is pushed.

We built four things to make that happen:

  • Waitlist emails via AWS SES — when someone submits their email, AWS sends them a confirmation and notifies you
  • Docker — packages the app into a portable, self-contained image that runs identically everywhere
  • AWS infrastructure via Terraform — the cloud resources that run the Docker container and serve it to the world
  • GitHub Actions CI/CD — automatically rebuilds and redeploys the site on every push to the main branch

2. Waitlist with AWS SES

SES stands for Simple Email Service — it is Amazon's managed email sending service. When a visitor submits their email on the landing page, two things happen simultaneously:

  1. A confirmation email is sent to the visitor
  2. A notification email is sent to usemoos.com@gmail.com so you know someone signed up

How it works technically

The form in WaitlistForm.tsx sends a POST request to /api/waitlist. That API route (a Next.js Route Handler) receives the email, validates it, and calls the AWS SES SDK to send both emails.

Browser form
    ↓ POST /api/waitlist { email }
Next.js Route Handler (app/api/waitlist/route.ts)
    ↓ SESClient.send()
AWS SES (eu-north-1)
    ↓ sends two emails
Confirmation → visitor's inbox
Notification → usemoos.com@gmail.com

SES sandbox vs production

Important: New AWS accounts start in SES sandbox mode. In sandbox mode you can only send emails to verified addresses. We requested production access — once approved by AWS (usually within a few hours), you can send to any address.

Sender identity

We verified the domain usemoos.com in SES, which allows sending from any address under that domain. The FROM address used is hello@usemoos.com.

Credentials in production

Locally, the app uses AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY from .env.local. In production on ECS, these are not needed — the ECS Task Role grants SES permission directly. The AWS SDK automatically detects this and uses the role credentials without any configuration change in the code.

3. Dockerizing the Next.js App

Docker packages the application and everything it needs to run (Node.js, dependencies, built files) into a single portable image. That image can then be run identically on any machine — your laptop, a cloud server, or an AWS container service.

Why Docker?

Without Docker, deploying a Node.js app to a server means installing Node, copying files, managing versions, and dealing with "it works on my machine" problems. With Docker, the image is the deployment artifact — the same thing that passes your local test is exactly what runs in production.

Multi-stage build

The Dockerfile uses three stages to keep the final image as small as possible:

StageWhat it doesWhy separate?
depsInstalls npm packagesCached separately so rebuilds skip this step if package.json hasn't changed
builderRuns next buildCompiles TypeScript, bundles CSS, generates static pages
runnerCopies only the built outputFinal image contains no source code or dev dependencies — just what's needed to run

Standalone output

We added output: "standalone" to next.config.ts. This tells Next.js to produce a minimal self-contained build in .next/standalone/ that includes only the files needed to run the server — no full node_modules folder. This is what makes the Docker image only 68MB instead of several hundred.

Running the container

# Build the image
docker build -t usemoose-marketing .

# Run it locally on port 3001
docker run -p 3001:3000 usemoose-marketing

The container runs as a non-root user (nextjs) on port 3000 for security.

4. AWS Infrastructure Overview

The diagram below shows how the pieces connect. A visitor's browser makes a request to usemoos.com and it travels through several AWS services before reaching the running application:

Browser ↓ Cloudflare DNS (resolves usemoos.com → CloudFront) ↓ AWS CloudFront (CDN — caches static assets globally) ↓ AWS ALB (Application Load Balancer — HTTPS termination) ↓ AWS ECS Fargate (runs the Docker container) ↓ Next.js app (port 3000)

Supporting services:

  • ECR — stores Docker images
  • ACM — provides SSL/TLS certificates for HTTPS
  • IAM — controls what the container is allowed to do (send emails via SES)
  • CloudWatch — stores logs from the running container

5. Each AWS Service Explained

ECR — Elastic Container Registry

ECR is AWS's private Docker image registry, like Docker Hub but private and inside your AWS account. When we build a new Docker image, we push it to ECR. When ECS needs to run the app, it pulls the image from ECR. We configured a lifecycle policy to keep only the last 10 images so storage costs don't grow unbounded.

ECS Fargate — Elastic Container Service

ECS Fargate runs Docker containers in the cloud without you managing any servers. You tell it what image to run, how much CPU and memory to allocate (256 CPU units / 512MB RAM for this app), and how many copies to run (1). AWS handles the underlying infrastructure — patching, scaling the host machines, replacing failed containers.

Key concepts:

TermWhat it is
ClusterA logical grouping of services — ours is named usemoose
Task DefinitionA blueprint describing the container: which image, how much CPU/memory, environment variables, ports
ServiceKeeps a specified number of task copies running at all times. If a container crashes, ECS restarts it automatically
TaskA running instance of the task definition — the actual container

ALB — Application Load Balancer

The ALB sits in front of ECS and handles incoming HTTP/HTTPS traffic. It does two things for us:

  • SSL termination — handles HTTPS using the ACM certificate, then forwards plain HTTP internally to the container
  • HTTP → HTTPS redirect — any request on port 80 is automatically redirected to port 443

The ALB uses a target group to know which containers to forward requests to. ECS automatically registers and deregisters containers in the target group as they start and stop.

CloudFront — Content Delivery Network

CloudFront is AWS's global CDN. It sits in front of the ALB and serves content from edge locations close to the visitor — so a user in Singapore doesn't have to wait for a response from Stockholm.

We configured three cache behaviours:

PathCache TTLWhy
/_next/static/*1 yearNext.js static assets have content hashes in filenames — safe to cache forever
/assets/*1 dayImages and icons change rarely
Everything else0 (no cache)Pages and API routes must be served fresh

On every deploy, GitHub Actions sends a CloudFront invalidation (/*) to flush the cache so visitors immediately see the new version.

ACM — AWS Certificate Manager

ACM provides free SSL/TLS certificates for HTTPS. We created two certificates:

  • One in eu-north-1 for the ALB
  • One in us-east-1 for CloudFront — CloudFront requires certificates to be in us-east-1 regardless of where your infrastructure is

Certificates are validated via DNS — we added CNAME records in Cloudflare that prove we own the domain. ACM renews certificates automatically before expiry.

IAM — Identity and Access Management

IAM controls permissions. We created two roles for the ECS task:

RoleUsed byPermissions
Execution RoleECS itself (to start the container)Pull image from ECR, write logs to CloudWatch
Task RoleThe running application codeSend emails via SES (ses:SendEmail)

The Task Role is what allows the waitlist API to send emails in production without hardcoded credentials. The AWS SDK automatically uses these role credentials when running inside ECS.

CloudWatch

CloudWatch collects logs from the running container. Every console.log and error in the Next.js app appears in CloudWatch under the log group /ecs/usemoose-marketing. Logs are retained for 30 days.

6. Terraform — Infrastructure as Code

Terraform is a tool that lets you describe infrastructure in code files (.tf files) and then create, update, or destroy it by running a command. Instead of clicking through the AWS console to create each resource, you write what you want and Terraform figures out how to create it.

Why this matters

Reproducibility: If something goes wrong and you need to recreate the infrastructure, you run terraform apply and everything is restored exactly as specified. No guesswork about what settings were used.

Key commands

CommandWhat it does
terraform initDownloads the AWS provider plugin — run once after cloning
terraform planShows what would be created/changed/destroyed without doing it
terraform applyCreates or updates resources to match the configuration
terraform destroyDeletes all resources managed by Terraform
terraform outputShows the output values (CloudFront ID, ECR URL, etc.)

File structure

FilePurpose
main.tfProvider configuration, VPC data sources
variables.tfConfigurable values: region, domain name, CPU/memory
ecr.tfDocker image registry
iam.tfExecution role and task role with SES permissions
alb.tfLoad balancer, security groups, HTTPS listener, ACM cert
ecs.tfCluster, task definition, service, CloudWatch log group
cloudfront.tfCDN distribution, cache behaviours, us-east-1 ACM cert
outputs.tfValues to print after apply: ECR URL, CloudFront ID, etc.

Two-phase apply for Cloudflare DNS validation

Because DNS is managed in Cloudflare (not Route 53), certificate validation requires a manual step. The apply was split into two phases:

1
Create only the ACM certificates: terraform apply -target=aws_acm_certificate.alb -target=aws_acm_certificate.cdn — this outputs the CNAME records needed for validation
2
Add those CNAME records in Cloudflare (DNS only, grey cloud)
3
Run full terraform apply — Terraform waits for ACM to confirm validation, then creates all remaining resources

7. CI/CD with GitHub Actions

CI/CD stands for Continuous Integration / Continuous Deployment. It means every time you push code to the main branch, an automated process builds and deploys the new version without you doing anything manually.

The workflow file

The file .github/workflows/deploy.yml defines what happens on every push to main. GitHub runs this automatically on their servers (called runners).

Steps in order

StepWhat it does
CheckoutDownloads the repository code onto the runner
Configure AWS credentialsAuthenticates with AWS using the deploy IAM user secrets stored in GitHub
Login to ECRAuthenticates Docker with the ECR registry so it can push images
Build & push imageBuilds the Docker image and pushes it with two tags: latest and the git commit SHA
Download task definitionFetches the current ECS task definition JSON from AWS
Render new task definitionUpdates the task definition with the new image URI
Deploy to ECSRegisters the new task definition and updates the ECS service — waits for stability
Invalidate CloudFrontClears the CDN cache so visitors get the new version immediately

GitHub Secrets

The workflow uses secrets stored in GitHub (Settings → Secrets → Actions) so credentials are never in the code:

SecretValue
AWS_ACCESS_KEY_IDDeploy IAM user access key
AWS_SECRET_ACCESS_KEYDeploy IAM user secret key
AWS_REGIONeu-north-1
ECR_REPOSITORYusemoose-marketing
ECS_CLUSTERusemoose
ECS_SERVICEusemoose-marketing
ECS_TASK_DEFINITIONusemoose-marketing
CONTAINER_NAMEusemoose-marketing
CLOUDFRONT_DISTRIBUTION_IDE1J6J326KTVC0I

Image tagging strategy

Every image is pushed with two tags: latest and the git commit SHA (e.g. sha256:abc123...). The SHA tag means every deploy is traceable — you can always identify exactly which commit is running in production and roll back to a previous image by deploying a specific SHA tag.

8. DNS with Cloudflare

The domain usemoos.com is registered on GoDaddy but DNS is managed through Cloudflare. Cloudflare is positioned in front of everything and is what visitors hit first when they type the domain.

Records added in Cloudflare

TypeNameTargetProxy
CNAME@d3te6mtqhkrat9.cloudfront.netDNS only (grey)
CNAMEwwwd3te6mtqhkrat9.cloudfront.netDNS only (grey)
CNAME_3ee56831d17d65e33bd25b599d013458ACM validation valueDNS only (grey)
CNAME_0ed03141cc250386d8c69c2dc261a1ed.wwwACM validation valueDNS only (grey)

Why DNS only (grey cloud)?

Cloudflare offers two modes — proxied (orange cloud) and DNS only (grey cloud). When proxied, traffic passes through Cloudflare's servers for DDoS protection and caching. We use DNS only because CloudFront already handles CDN and caching. Putting Cloudflare proxy in front of CloudFront causes SSL conflicts and adds unnecessary complexity.

Note: The ACM validation CNAME records must always remain in Cloudflare. ACM uses them to automatically renew the certificates annually. Deleting them would cause the certificates to expire.

9. How a Request Flows End to End

Here is what happens when someone visits usemoos.com:

1
DNS resolution — the browser asks Cloudflare's DNS where usemoos.com is. Cloudflare returns the CloudFront distribution domain (d3te6mtqhkrat9.cloudfront.net)
2
CloudFront edge — the request hits the nearest CloudFront edge location. For a static asset like a CSS file, CloudFront returns it directly from cache. For a page or API request, it forwards to the origin
3
ALB — CloudFront forwards to the ALB in Stockholm (eu-north-1). The ALB terminates HTTPS and forwards plain HTTP to the ECS container on port 3000
4
ECS container — the Next.js app handles the request and returns a response
5
Response travels back through ALB → CloudFront → browser

Waitlist form submission

1
Form submits POST to /api/waitlist
2
CloudFront passes it through (no cache on API routes)
3
Next.js route handler validates the email and calls SES
4
SES sends confirmation to visitor and notification to usemoos.com@gmail.com
5
Route handler returns { ok: true }, form shows success state

11. What Happens on Every Deploy

Once everything is set up, the deployment process is fully automatic. Here is the complete lifecycle of a code change:

1
You edit code and push to the main branch on GitHub
2
GitHub detects the push and triggers the deploy.yml workflow on a GitHub-hosted runner
3
The runner builds a new Docker image from the latest code
4
The image is pushed to ECR tagged with the commit SHA
5
A new ECS task definition revision is registered pointing to the new image
6
ECS starts a new container with the new image (old container keeps running until new one is healthy)
7
Once the new container passes health checks, ECS stops the old one
8
CloudFront cache is invalidated — visitors get the new version on next request
9
GitHub Actions reports success — total time approximately 3-5 minutes
